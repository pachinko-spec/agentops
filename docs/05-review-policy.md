# レビュー方針

## 目的

レビューは、バグ、仕様逸脱、セキュリティリスク、テスト不足、ドキュメント乖離を見つけるために行う。

## 複数視点レビュー

高リスク変更では、視点を分けたサブエージェントを使う。

- correctness: 仕様通りか
- security: 認可、秘密情報、入力検証、依存関係
- tests: テスト不足、flaky、E2E要否
- docs: README、設計docs、運用手順との整合
- maintainability: 過剰抽象化、既存パターン逸脱

## cross-review

cross-review / cross-model review は、主 orchestrator とは別系列、別 CLI、別モデルファミリーの frontier reviewer で設計や差分を確認するレビューである。特定モデル固定ではなく、現在利用できる CLI、公式 docs で確認済みの model id、対象タスクのリスクに応じて選ぶ。

5 工程フローの実装レビューでは、4-α 同系列独立実装レビューと 4-β cross-review を分ける。4-α は実装担当と同系列でも別 session の独立検証であり、cross-review ではない。4-β は実装担当と別モデルファミリーの `review_frontier` が行う本来の cross-review とする。

次の変更では cross-review を検討する。

- 高リスク設計、セキュリティ修正、データ損失リスクがある変更
- 新規機能追加、リファクタリング、依存追加
- API 契約変更、デプロイ影響、本番運用への影響
- レビュー指摘を修正した後の再確認

cross-review を実施しても、指摘の採否、修正範囲、延期、統合判断は主 orchestrator が持つ。小さな docs 修正や低リスクの機械的変更では省略してよいが、省略理由が重要な場合は PR / handoff / run log に残す。

## 指摘優先度

| 優先度 | 意味 | 対応 |
| --- | --- | --- |
| P0 | 即時停止級。データ損失、重大セキュリティ、ビルド不能 | 必ず修正。無理ならユーザー確認 |
| P1 | マージ前に必須修正 | 必ず修正 |
| P2 | 修正推奨。延期可能だが理由が必要 | 修正または次セッションへ記録 |
| P3 | nit、改善案、将来検討 | 原則Issue化しない |

## ループ防止

- レビュー修正は最大2周。
- レビュー後に修正した場合は必ず再レビューし、作業を修正で終わらせない。
- 3周目が必要な場合は、主 orchestrator が統合判断を行う。
- P3だけで修正ループを続けない。
- AIが自律的に大量のIssueを起票しない。
- unresolved P0/P1 が残る場合はマージしない。
- ループ終了時は、最終レビューで未解決の P0/P1 がなく、P2 の修正または延期理由が明示されていることを確認する。

## 最終判断

サブエージェントは所見を出す。採否、修正範囲、延期判断、マージ可否は主 orchestrator が責任を持つ。最終応答またはマージ前判断では、最後に行ったレビュー結果を明示し、修正直後の未レビュー状態を残さない。
