## 確認観点（Round 1 投入時に message に含める）

1. **version pin**: `actions/checkout@v4` (tag pin)、`actionlint` の SHA pin via download script（commit `914e7df21a07ef503a81201c76d2b11c789d3fca`）、`gaurav-nelson/github-action-markdown-link-check@3c3b66f1f7d0900e37b71eca45b63ea9eedfce31`（commit SHA pin）。supply-chain 観点で SHA pin 漏れがないか。
2. **fail / warn の責務分離**: actionlint / yamllint / markdown-link-check の 3 fail 系と freshness-check の 1 warn 系。warn 系は `continue-on-error: true` ではなく **`exit 0` 明示** で fail を抑止。required check 設定 / branch protection との相互作用で意図せず block する pattern がないか。
3. **freshness-check grep の YAML frontmatter 専用化**: task 04 Round 1 P3 申し送りで「`^last_reviewed|^> last-reviewed` の OR で旧 blockquote 形式を許容しない」が反映されているか。`^next_review_by:` 単独で正規化されているか。
4. **markdown-link-check の rate limit 対策**: `retryOn429: true` / `retryCount: 3` / `fallbackRetryDelay: "30s"` / `timeout: "20s"` / `aliveStatusCodes: [200, 206, 999]` / `ignorePatterns` (localhost / example.com / GitHub issues / PRs) が GitHub の rate limit + 外部 URL 一時失敗を捌けるか。false positive 頻度が運用に耐えるか。
5. **`.gitignore` 追記の妥当性**: 既存 26 行と機能的衝突していないか。`wrangler.toml` を ignore しない注記が将来 secret 流出を誘発する可能性がないか（コメント文面で運用ミスを誘発しないか）。
6. **`.dev.vars` の用途明示**: Wrangler local dev secrets として明示しているが、他用途で同名ファイルを使用するワークフローと衝突する pattern がないか。
7. **PR template の DbC 整合**: docs/03-dbc-and-quality-gates.md の DbC 5 条件（前提 / 不変 / 完了 / 禁止 / 停止）と template の checklist が対応しているか。
8. **CI 実行コスト試算**: 上記表が public repo の無料枠と private repo の 2,000 分 / 月制限の両方で十分なマージンか。markdown-link-check の rate limit ヒット時の最悪 run 時間概算が妥当か。
9. **既存 hook との責務分離**: `scripts/hooks/pre-commit` / `pre-push` と CI が責務を混同していないか。二重実行による無駄がないか。
